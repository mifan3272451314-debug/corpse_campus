- 本文将“异常特性”定义为一套基于 **Iron's Spellbooks 原生法术书与书槽** 的异常序列系统，而不是普通装备物品。它承担 **身份绑定、法术搭载、序列晋升、世界唯一性控制、运维指令** 五类职责。

  **依据**：`能力列表.md` 最新版五流派分层、`ModSpells.java` 当前注册结果、`ModSchools.java` 当前五流派注册结果。

- 你已额外确认以下关键边界：  
  1. **沿用原生法术书与书槽**；  
  2. **首次吞噬后永久锁流派**；

## 3. 行业与业务场景定位

### 3.1 当前业务场景

当前模组不是传统纯 PVE 魔法模组，而是更接近 **多人服异能争夺 / 序列晋升 / 身份稀缺玩法**。在这种业务场景下：

- 异能不是“你学了什么技能”，而是“你持有什么异常资格”；
- 法术书不是普通容器，而是玩家资格的载体；
- S 级唯一和 40 上限不是平衡项，而是世界生态控制项；
- 运维命令不是可选辅助，而是必需功能。

**依据**：原始需求第 2、6、20、21 条；多人服稀缺资格系统的一般设计规律。

### 3.2 本系统的产品定位

“异常特性”应被定义为：

> **绑定在玩家身上的异常身份核心物品**，它以原生法术书为表现层，以法术搭载为能力层，以序列与世界限制为规则层。

**依据**：原始需求第 1、2、6、7、8、14、20、21 条。

## 7. 核心规则总则

### 7.1 异常法术书规则

1. 玩家必须始终拥有一本文档定义的异常法术书；
2. 法术书默认不提供基础法力；
3. 法术书不可摘下；
4. 法术书绑定主人 UUID；
5. 法术书中的异常法术决定法力加成与流派强化；
6. 法术书是异常资格的唯一合法载体。

**依据**：原始需求第 1、2、3、4、6、7 条。

### 7.2 玩家规则

1. 第一次进入世界时必须被发放并强制佩戴异常法术书；
2. 若玩家丢失异常法术书，系统必须补发或重建；除非游戏内拥有异常特性书的玩家超过了设定数量
3. 玩家第一次获得 B 级异常特性后永久锁定主序列；
4. 成为异常者后默认不能再获得其他流派异常特性；
5. 唯一例外是圣祈 A 级技能“嫁接师”；
6. 玩家死亡后，法术书掉落但仍绑定原主人；
7. 非主人将书拿进物品栏后，法术书自动消失。

#### 关于异常特性吞噬物的新增规则补充

8. 玩家死亡时，异常特性吞噬物按照“历史阶段”掉落，而不是只掉落当前最高阶段；
9. 每条路线按当前最高已解锁等级向下展开全部历史阶段：`B`、`B+A`、`B+A+S`；
10. 每个掉落物会记录前持有者与其生前异能信息；
11. 每个阶段物品 tooltip 必须显示该阶段包含的异能名称。

**依据**：原始需求第 2、6、14、16、18 条；已确认的特殊口径。

### 7.3 世界规则

1. 每条序列的 S 级同一时间只能有一个持有者；
2. 全局异常者数量 + 玩家背包中的异常法术书数量，总和不得超过 40；
3. 达到 40 后，所有与 B 级异常特性相关的产出全部禁用；
4. 回溯之虫触发时，按“施法者穿越到镜像维度中的备份时刻”逻辑处理，而不是直接覆盖当前主维度；
5. 生生不息一旦施放完成，整个世界永久封禁该 S 级资格再次出现。

**依据**：原始需求第 20、21 条；已确认特殊口径。

## 8. 数据结构设计

## 8.1 总体设计原则

建议采用三层数据源：

1. **书物品 NBT**：记录书自身身份与绑定关系；
2. **玩家持久化数据**：记录玩家异常身份与解锁状态；
3. **世界 SavedData**：记录全局唯一席位、世界计数与封号状态。

**结论**：不要把所有信息只塞进书 NBT，也不要只塞进玩家 NBT，必须三层同时存在。  
**依据**：书会掉落、玩家会死亡、世界要统计 40 上限与 S 唯一，这三个维度职责不同。

### 8.2 书物品 NBT 字段

建议前缀统一为 `Anomaly*`。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `AnomalyBookId` | UUID | 这本异常书的唯一 ID |
| `AnomalyOwnerUuid` | UUID | 绑定主人 UUID |
| `AnomalyOwnerName` | String | 主人名称快照 |
| `AnomalyMainSequence` | String | 主序列：`xujing/rizhao/dongyue/yuzhe/shengqi` |
| `AnomalyHighestRank` | String | 当前最高阶：`NONE/B/A/S` |
| `AnomalyLocked` | boolean | 是否不可摘下 |
| `AnomalyIsRebuilt` | boolean | 是否由系统补发 / 重建 |
| `AnomalyGrafterUnlocked` | boolean | 是否已获得嫁接师权限 |
| `AnomalyShengshengConsumed` | boolean | 是否已因施放生生不息而封号 |
| `AnomalyManaCache` | int | 当前法力加成缓存 |
| `AnomalySchoolBonusCache` | CompoundTag | 各流派强化缓存 |
| `AnomalyLastRecalcGameTime` | long | 上次重算时间 |

**说明**：书中实际搭载的法术列表，优先使用 Iron's Spellbooks 原生法术容器作为权威数据源，异常系统不重复维护第二套完整法术列表，只维护缓存与异常元数据。  
**依据**：沿用原生 `SpellBook` 的设计前提；避免重复存储导致数据不一致。

### 8.2.1 当前异常特性吞噬物 ItemStack 附加字段

为满足“死亡后掉落历史阶段物品，并在物品上写明之前拥有的异能”这一业务要求，当前实现会在异常特性吞噬物的 `ItemStack` 上附加以下 NBT：

**说明**：这些字段不用于战斗逻辑判定，主要用于掉落展示、剧情交互与后续吞噬/催化判定。  
**依据**：当前 `AnomalyTraitItem` 的 tooltip 逻辑与本次新增业务需求。

### 8.2.2 当前历史阶段生成规则

为了保证掉落物能够表达“之前拥有着什么样的异能”，当前实现按 **路线维度** 统计玩家异常法术书中已装载法术，再按 **该路线历史阶段** 生成物品：

1. 先读取异常法术书中的已装载法术；
2. 按路线聚合；
3. 取该路线已达到的最高阶段；
4. 按最高阶段向下生成全部历史阶段物品；
5. 对每个阶段分别写入该阶段异能名称；
6. 同时把所有路线能力汇总写入“其生前异能”。

**结论**：当前实现是“按路线保留历史阶段”，不是“只掉一件当前最高级物品”。  
**依据**：最新代码实现口径与本次需求变更。

## 9. 数值规则与公式

### 9.1 法术等级对应的异常等级

- 两个字的异能 = 一级异常 = B 级；
- 三个字的异能 = 二级异常 = A 级；
- 四个字的异能 = 三级异常 = S 级。

**说明**：这里指的是你的异常设定层级，不是 Iron's Spellbooks 自带的法术稀有度。  
**依据**：原始需求第 3 条；`能力列表.md` 当前按一级 / 二级 / 三级分类。

### 9.2 法力加成公式

#### 默认规则

- B 级已搭载法术：每个 `+100` 法力
- A 级已搭载法术：每个 `+150` 法力
- S 级已搭载法术：每个 `+200` 法力
- 异常法术书基础法力：`0`

#### 公式

```text
ManaBonus = BCount * 75 + ACount * 150 + SCount * 250
```

#### 统计口径

- 按 **已搭载技能数** 计算；
- 不是按“当前激活数量”计算；
- 不是按“当前冷却结束数量”计算；
- 不是按“主序列数量”单独计算。

**依据**：原始需求第 3、4、5 条；已确认“按已搭载技能数计入”。

### 9.3 流派强化公式

#### 默认规则

- B 级法术：对其所属流派 `+25` 强化
- A 级法术：对其所属流派 `+50` 强化
- S 级法术：对其所属流派 `+50` 强化

#### 公式

```text
SchoolBonus[school] = BCountOfSchool * 25 + ACountOfSchool * 50 + SCountOfSchool * 50
```

#### 实现建议

- 默认把“流派强化”映射为 **对应 School 的 Spell Power 百分比加成**；
- 若后续测试发现过强，可通过配置切换成原始属性点加成。

**依据**：原始需求第 7 条；`ModSchools.java` 当前已存在 SchoolType 与法术伤害体系；采用 School Power 是最直接且与原模组最兼容的实现。

### 9.4 法术重复添加规则

默认建议：

1. 同一本异常书 **不允许完全重复添加同一法术**；
2. 如果命令再次添加同一法术，默认行为是：
   - 若 `spellLevel` 高于当前等级，则升级到更高等级；
   - 若不高于当前等级，则本次添加视为跳过；
3. `/magic add` 中的 `count` 表示重复尝试次数，多余次数记为跳过数。

**依据**：原始命令格式保留了“添加数量”字段，但原生法术书通常不需要重复占用多个同法术槽位，因此采用“重复尝试 / 升级优先”是兼顾兼容性与命令语义的最稳方案。

### 9.5 日兆的当前能量机制

#### 已确认规则

- 日兆序列 **初始当前能量 = 0**；
- 当玩家处于 **白天 + 头顶无遮挡** 时，当前能量缓慢恢复；
- 这里恢复的是 **当前能量**，不是能量上限；
- 日轮金乌等技能会直接吃掉当前能量。

#### 默认建议数值

- `rizhao.regenPerSecond = 2.0`
- `rizhao.requireDaytime = true`
- `rizhao.requireOpenSky = true`
- `rizhao.pauseWhileGoldenCrowExists = true`

**依据**：最新版 `能力列表.md` 的日兆资源机制说明；已确认“恢复的是当前能量，不是上限”。

## 10. 玩家与世界流程设计

## 10.1 首次进入世界流程

### 目标

保证玩家第一次进入世界时，必然拥有并佩戴异常法术书。

### 执行步骤

1. 监听玩家首次登录事件；
2. 检查玩家持久化数据中 `HasAnomalyBook`；
3. 若为 `false`：
   - 创建一本文档定义的异常法术书；
   - 写入 `OwnerUuid / OwnerName / BookId / Locked=true`；
   - 若书槽已有其他书，优先将原书移入背包第一空位；
   - 若背包已满，则将原书投放到脚下；
   - 强制把异常书放入书槽；
   - 将玩家 `HasAnomalyBook=true`，写入 `BoundAnomalyBookId`；
4. 向玩家发送一次性系统提示。

### 结论

- “首次进入世界强制佩戴”必须是 **服务端权威流程**，不能依赖客户端界面操作。  
  **依据**：原始需求第 2 条；多人服环境下客户端不可被信任。

### 10.2 缺书补发 / 自动修复流程

### 触发条件

- 玩家登录时没有绑定书；
- 玩家重生后没有绑定书；
- 玩家切维度后书槽数据异常；
- 玩家背包、末影箱、书槽中都找不到合法绑定书。

### 执行步骤

1. 从玩家持久化数据读取 `LastValidBookSnapshot`；
2. 若存在快照，则按快照重建异常书；
3. 若不存在快照，则按玩家已解锁异常状态重建一本文档定义的新书；
4. 重新写入 `BoundAnomalyBookId`；
5. 强制放回书槽并同步客户端。

**依据**：原始需求第 2、6 条；绑定物品系统必须具备断档自愈能力。

### 10.3 右键吞噬 B 级异常特性流程

### 触发前提

- 玩家已佩戴异常法术书；
- 当前书中没有任何异常法术；
- 玩家 `AnomalyAwakened=false`；
- 主手或副手持有某个 `trait_<sequence>_b`。

### 执行步骤

1. 识别被吞噬物所属序列；
2. 从该序列 B 池中随机抽取 1 个尚未拥有的 B 级法术；
3. 将该法术添加到异常法术书；
4. 写入玩家主序列 `AnomalyMainSequence`；
5. 写入 `AnomalyAwakened=true`；
6. 写入 `AnomalyHighestRank=B`；
7. 消耗 1 个 B 级异常特性物品；
8. 重新计算法力加成与流派强化。

### 结论

- 玩家第一次吞噬实际上是“觉醒 + 锁定主序列 + 获得 1 个 B 技能”的合并流程。  
  **依据**：原始需求第 14、16 条。

### 10.4 锁流派规则

默认规则如下：

1. 一旦玩家已觉醒，后续吞噬行为只能接受 **同主序列** 的异常特性；
2. 任何跨序列吞噬默认失败；
3. 唯一例外：当玩家已拥有圣祈 A 级“嫁接师”后，可开启跨序列吸收权限；
4. 即使拥有嫁接师，也 **不能绕过 S 级唯一性**，也 **不能绕过 40 上限**。

**依据**：原始需求第 16 条；已确认“嫁接师”是唯一例外。

### 10.5 B→A 晋升流程

### 触发条件

- 玩家主序列已锁定；
- 玩家当前最高阶为 `B`；
- 玩家持有同主序列的 `trait_<sequence>_b`；
- 完成该序列对应的 B→A 仪式。

### 执行步骤

1. 校验玩家当前是否已拥有至少 1 个本序列 B 技能；
2. 校验玩家持有同序列 B 级异常特性；
3. 校验仪式条件；
4. 从该序列 A 池中随机抽取 1 个尚未拥有的 A 级法术；
5. 将法术写入书中；
6. 更新 `AnomalyHighestRank=A`；
7. 消耗 1 个 `trait_<sequence>_b`；
8. 重算法力与流派强化。

**依据**：原始需求第 18 条明确要求“吞噬自身序列的同级别异常特性并完成特定仪式晋升到更高序列”。

### 10.6 A→S 晋升流程

### 触发条件

- 玩家当前最高阶为 `A`；
- 玩家持有同主序列的 `trait_<sequence>_a`；
- 对应序列当前没有其他 S 持有者；
- 完成该序列 A→S 仪式。

### 执行步骤

1. 校验 序列 S 唯一席位是否为空；
2. 校验玩家持有同序列 A 级异常特性；
3. 校验仪式完成；
4. 添加该序列唯一的 S 级法术；
5. 把玩家 UUID 与书 UUID 写入 `SequenceSUniqueOwners`；
6. 更新 `AnomalyHighestRank=S`；
7. 消耗 1 个 `trait_<sequence>_a`；
8. 重算法力与流派强化。

**依据**：原始需求第 18、20 条；每条序列 S 级只有一个法术且只能有一个持有者。

### 10.7 死亡绑定与掉落流程

### 默认规则

1. 玩家死亡时，异常书允许掉落；
2. 书掉落后仍然保留原主人 UUID；
3. 非主人拾取后，只要进入对方背包就立即自毁；
4. 主人若未及时捡回，系统在重生或登录时可补发 / 重建一份合法书；
5. 原掉落书若随后又被其他人捡到，仍然自毁，不会和重建书冲突。

### 结论

- 死亡绑定应当做成“掉落可见，但持有权限受限”的模型，而不是“死亡不掉落”。  
  **依据**：原始需求第 6 条写的是“死亡掉落，别人拿在物品栏上会自动消失”。

### 10.8 非法持有检查流程

建议检查节点：

1. 玩家拾取物品事件；
2. 玩家背包变动事件；
3. 玩家登录事件；
4. 玩家每秒背包巡检一次；
5. 容器关闭回背包时再次校验。

### 非法持有判定

- 物品为异常法术书；
- `AnomalyOwnerUuid != 当前玩家 UUID`。

### 处理结果

1. 播放异常湮灭粒子 / 音效；
2. 删除该物品；
3. 记录日志；
4. 可选向持有者提示“该异常特性与你不兼容，已湮灭”。

**依据**：原始需求第 6 条；多人服必须考虑背包绕过、丢拾取与复活捡包等情况。

### 10.9 40 上限检测流程

### 已确认统计口径

`全局异常数量 = 当前异常者人数 + 当前玩家背包中的异常法术书数量`

### 默认重算时机

1. 玩家登录 / 退出；
2. 玩家觉醒 / 失去异常资格；
3. 玩家背包中异常法术书数量变化；
4. 服务器启动完成后全量扫描一次；
5. 管理员执行 `/magic limit recount` 时手动重算。

### 达到 40 后的处理

1. 禁用全部 B 级异常特性配方；
2. 禁用全部 B 级异常特性自然掉落；
3. 禁用全部 B 级异常特性任务奖励；
4. 管理员命令仍可通过 `force` 参数绕过，但默认关闭；
5. 已存在的异常者与书不受影响。

**依据**：原始需求第 21 条；已确认“异常特性总数统计玩家背包法术书”。

## 11. 默认仪式方案（可配置）

> 说明：原始需求只要求“完成特定仪式”，未给出具体仪式内容。为了让本文成为可执行文档，以下给出一套 **默认仪式建议**，后续可通过配置改动。

### 11.1 B→A 仪式默认表

## 12. 特殊例外规则专章

### 12.1 圣祈 A 级：嫁接师

#### 已确认规则

- 嫁接师是“永久锁流派”的唯一例外；
- 拥有嫁接师后，玩家可以吸收其他异能；
- 玩家还可以把自己的任意异能嫁接给无异能者。

#### 实现建议

1. 嫁接师只解除“跨序列法术吸收限制”，不解除主序列身份；
2. 玩家 `AnomalyMainSequence` 仍保持原主序列不变；
3. 嫁接师额外吸收的跨序列法术，应当按其 **真实所属流派** 计入法力与流派强化；
4. 若跨序列获得的是 S 级法术，仍然要占用对应序列的 S 唯一席位；
5. 嫁接给“无异能者”时，默认目标必须 `AnomalyAwakened=false`。

#### 结论

- 嫁接师是“跨序列白名单”，不是“无视所有世界规则的超级权限”。  
  **依据**：你已确认它只是锁流派例外；原始需求并未授权其绕过唯一性与全局上限。

### 12.2 圣祈 S 级：生生不息

#### 已确认规则

- 生生不息在 **施放后** 才会永久封号；
- 施放后，自身会获得圣祈全部一阶与二阶能力；
- 同时会失去“生生不息”本身；
- 此后全服不能再次获得“生生不息”。

#### 实现步骤

1. 施法前校验 `EndlessLifeSealed=false`；
2. 施法成功后复活所有死亡玩家；
3. 给施法者补齐圣祈全部 B / A 法术；
4. 从施法者书中移除 `endless_life`；
5. 将世界数据 `EndlessLifeSealed=true`；
6. 将玩家与书上的 `AnomalyShengshengConsumed=true`；
7. 后续任何获取 / 添加 / 掉落 / 命令发放 `endless_life` 的流程全部拒绝。

#### 结论

- 生生不息的唯一性判断点是 **施放成功瞬间**，不是第一次获得瞬间。  
  **依据**：已确认口径；`能力列表.md` 文本“施法后会失去生生不息本身，之后无法再有人获得”。

### 12.3 虚境 S 级：回溯之虫

#### 已确认规则

- 回溯之虫不是“把当前主维度整体回写成旧状态”，而是 **使用后穿越到镜像维度中的备份时刻**；
- 不能简化成 30 秒回档技能；
- 也不能再按“当前世界就地恢复覆盖”的方案理解。

#### 设计建议

由于 Minecraft 不是天然支持“真正的时间回退”，因此实际落地建议做成 **镜像备份维度穿越模型**：

1. 为每个受支持的主维度建立一个镜像备份维度；
2. 定期或在关键节点把主维度状态同步到镜像维度；
3. 每次备份完成后，记录该镜像维度对应的备份时间戳、源维度 ID、镜像维度 ID 与安全出生点；
4. 施放回溯之虫时，不覆盖当前主维度，而是将施法者传送到对应镜像维度中“那个已经备份好的时刻”；
5. 如后续玩法需要，也可以扩展为把指定目标或一组玩家一起送入镜像备份维度，但默认先按“施法者进入镜像备份维度”实现；
6. 镜像维度内的区块、实体、方块实体与世界数据，均以备份时刻的快照为准；
7. 若镜像维度缺失、备份损坏、目标位置无效，则施法失败并把玩家回退到安全点。

#### 默认实现拆分

- **P0**：只支持主世界 + 下界 + 末地三维度镜像，且默认只有施法者自己穿越；
- **P1**：支持其他自定义维度与指定目标同行；
- **P2**：支持外部模组维度与区块级增量快照。

#### 风险提示

这是本系统里开发成本最高、对存档安全要求最高的能力，必须单独测试，不应与普通技能同优先级并行开发。

**依据**：已确认口径；`能力列表.md` 文本已经从“时间倒流”改写为“世界备份切换”，且你进一步确认了“使用之后会穿越到镜像维度备份的时候”的表现方式。

### 13.6 `/magic add` 完整语法说明

#### 固定语法

```text
/magic add <playerOrEntityId> <spellId> <spellLevel> <count>
```

#### 参数说明

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `playerOrEntityId` | `String / int` | 玩家名或在线实体 ID，最终必须解析到合法玩家 |
| `spellId` | `String` | 注册 ID、中文别名或序列索引别名 |
| `spellLevel` | `int` | 目标法术等级，建议 1~法术上限 |
| `count` | `int` | 添加尝试次数，默认用于批量升级 / 重复尝试 |

#### 参数校验顺序

1. 解析目标玩家；
2. 校验目标玩家在线；
3. 校验目标玩家当前有合法异常书；
4. 解析 `spellId` 到最终注册 ID；
5. 校验法术存在于 **当前已注册法术** 或 **允许的待实现白名单**；
6. 校验 `spellLevel >= 1`；
7. 若配置要求，则校验 `spellLevel <= 该法术最大等级`；
8. 校验 `count >= 1` 且不超过 `command.maxBatchCount`；
9. 校验序列是否匹配；
10. 若为跨序列，则校验目标玩家是否有嫁接师权限；
11. 若添加的是 S 级法术，则校验唯一席位与封号状态；
12. 成功后执行添加并重算法力 / 强化。

#### 建议返回值

- `successCount`：实际成功添加 / 升级次数
- `skipCount`：因重复或低等级被跳过的次数
- `failReason`：失败原因码

**依据**：原始需求固定给出了 `/magic add` 语法；多人服需要更完整的失败可观测性。

## 16. 测试与验收清单

### 16.1 单人基础测试

1. 新玩家首次进入世界，是否自动获得并佩戴异常书；

3. 无异常者吞噬 B 级异常特性后，是否锁定主序列并获得 B 法术；
4. 书中法术变化后，法力与流派强化是否立即重算。

### 16.2 多人服绑定测试

1. 玩家 A 死亡后，书掉落是否保留绑定；
2. 玩家 B 捡到 A 的书后，是否进背包即消失；
3. 玩家 A 重生 / 登录后，是否能正确补回书；
4. 重建书是否与玩家快照保持一致。

3. 非法法术 ID 是否正确报错；
4. S 级唯一占用时是否拒绝添加；
5. 生生不息封号后是否彻底拒绝添加。

### 16.4 世界规则测试

## 18. 可能遗漏的关键点

### 18.1 仍需在实现阶段进一步锁死的细节

1. **异常书的“不可摘下”具体拦截点**：是直接阻止槽位点击，还是允许拖下后立即回弹；
2. **玩家背包统计范围**：是否包含副手、快捷栏、主背包、末影箱；当前文档默认只统计“玩家当前背包 / 装备栏中的异常书”；
3. **未实现法术的命令白名单**：是允许先配置后占位，还是必须等代码注册完才能命令添加；
4. **日兆当前能量与 Iron's Spellbooks 法力池的换算关系**：是否完全复用同一资源，还是增加独立的当前能量缓存；
5. **嫁接师跨序列赠与后的归属**：目标获得的法术是否立刻锁定为目标主序列、是否影响唯一席位，需要在代码实现时彻底写死。

**依据**：这些点会直接影响系统是否会出现复制、穿透规则或数值混乱。

### 18.2 推荐的最终实现裁定

为了减少歧义，建议实现时按以下口径最终落死：

1. 不可摘下 = **服务端取消摘下行为 + 每秒兜底回填**；
2. 背包统计 = **玩家主背包 + 快捷栏 + 装备栏**，默认不含末影箱；
3. 未实现法术 = **允许命令白名单占位，但默认不开启施放**；
4. 日兆当前能量 = **优先复用当前法力值显示层，但保留单独恢复规则**；
5. 嫁接师跨序列赠与 = **不改目标主序列，只增加额外法术访问权**。

**依据**：这是兼顾当前代码基础、实现复杂度与后续维护成本的最稳方案。

## 19. 原始需求附录

以下为本文件展开前的原始需求核心点，保留作追溯依据：

1. 需要制作一个法术书，名字叫做异常特性，无法被摘下；
2. 需要添加指令，若玩家第一次进入世界或当前还没有异常特性，就被强制佩戴该法术书；
3. 法术书默认不提供法力值；
4. 法术书会根据里面添加的异能法术增加法力值并强化流派；
5. 一级增加 75 法力，二级增加 150 法力，三级增加 250 法力；
6. 法术书有绑定，玩家死亡掉落后，别人拿在物品栏上会自动消失；
7. 法术书会检测搭载法术流派并强化，一级增加 25，对应二级增加 50，三级增加 50；
8. 需要命令检测注册的所有法术，并通过命令给当前法术书添加对应法术；
9. 指令格式：`/magic add 生物ID或者玩家名称 法术ID 法术等级 添加数量`；
10. 异常分五条序列三个等级：B、A、S；
11. 每个序列 B 级四种、A 级两种、S 级一种；
12. 无异常者右键吞噬一级异常特性后成为 B 级异常；
13. 成为异常者后无法再吞噬其他异常特性；
14. 异常者可通过吞噬自身序列同级异常特性并完成仪式晋升；
15. S 级异常者每条途径只能同时存在一位；
16. 全场异常者与异常特性总和上限为 40，达到后 B 级异常特性停止产出。

**依据**：最初需求原文。
