## 1. P0 · Critical（阻塞性 bug）

### 1.1 `impermanence_monk`（东岳 A 级·无常僧）注册缺失

**现象**：

- 法术主类 [`ImpermanenceMonkSpell.java`](src/main/java/com/mifan/spell/dongyue/ImpermanenceMonkSpell.java) 存在且完整
- 运行时 [`ImpermanenceMonkRuntime.java`](src/main/java/com/mifan/spell/runtime/ImpermanenceMonkRuntime.java) 存在
- `ModMobEffects` 已注册 `IMPERMANENCE_MONK` / `IMPERMANENCE_MONK_INFECTED` 两条标记效果
- `AbilityEventHandler#onLivingAttack` 已挂被感染者伤害阻断逻辑
- `AnomalyBookService.SPELL_SPECS` 已登记 `impermanence_monk → (无常僧, A, 东岳)`
- `zh_cn.json` / `en_us.json` 含完整 24 条相关翻译
- 图标 `impermanence_monk.png` 已存在

但 [`ModSpells.java`](src/main/java/com/mifan/registry/ModSpells.java) 全文 grep 不到任何 `"impermanence_monk"` 字符串，也没有 `IMPERMANENCE_MONK = SPELLS.register(...)` 行。

**后果**：ISS 的 `irons_spellbooks:spells` 注册表里根本没有这个法术，`/magic add <player> impermanence_monk 1 1` 返回 `ERR_SPELL_UNKNOWN`；玩家即使吞了东岳 A 级特性也永远抽不到「无常僧」；`AnomalyBookService.SPELL_SPECS` 的对应条目实际上是"空指针条目"。

**修复**（一行）：

```java
// 在 ModSpells.java 东岳 A 级法术段内新增
public static final RegistryObject<AbstractSpell> IMPERMANENCE_MONK =
        SPELLS.register("impermanence_monk", ImpermanenceMonkSpell::new);
```

**验证**：

1. `./gradlew compileJava` 不应报错
2. 启动 client，`/magic lookup 无常僧` 应返回 `状态: 已注册`
3. `/magic add @s impermanence_monk 1 1` 可写入
4. `/magic list spells dongyue` 列表中应能看到它

## 3. P2 · Medium（代码质量 / 安全 / 设计一致性）

### 3.1 MobEffect 分类错误

`ModMobEffects.java` 几乎所有 effect 都是 `MobEffectCategory.BENEFICIAL`（只有 `GoldenCrowStunEffect` 单独声明为 HARMFUL）。其中：

| 效果 | 当前分类 | 建议 | 理由 |
| --- | --- | --- | --- |
| `MANIA` | BENEFICIAL | NEUTRAL 或 HARMFUL | 狂躁是"被副作用支配"，不是单纯增益 |
| `NECROTIC_UNDEAD` | BENEFICIAL | NEUTRAL | 丧尸化带副作用，不应被治疗药水净化 |
| `IMPERMANENCE_MONK_INFECTED` | BENEFICIAL | HARMFUL | 这是被感染者身上的负面标记 |
| `SUNLIGHT_NEUTRALIZED` | BENEFICIAL | NEUTRAL | 丧尸被中立化，本来对丧尸而言不算"益" |

影响：`MilkBucketItem.finishUsingItem` 清 `HARMFUL` 效果；治疗药水清 `BENEFICIAL` 效果；信标治疗反馈。分类错了，玩家操作结果就不符合直觉。

### 3.2 Serverbound 包校验空档

扫描 `src/main/java/com/mifan/network/serverbound/` 下所有包的 `handle()`：

- `SetMidasTouchTimerPacket`：`powerLevel` / `timerSeconds` / `spellLevel` 全信任客户端；`armFromPlayerSelection` 只 clamp 了 timerSeconds
- `SetDominanceTargetPacket`：传入 targetUuid，无距离校验、无施法权限校验
- `SetRecorderOfficerTimerPacket`：`targetEntityId` / `timerSeconds` 无距离与合法性校验
- `MimicAbsorbPacket` / `MimicReleasePacket`：slot 索引、目标 UUID 无范围保护
- `ConfigureDesignatedAbilityPacket` / `SaveAdminConfigPacket` / `AdminExecuteCommandPacket`：需要严格 `sender.hasPermissions(4)` 门禁，尤其 `AdminExecuteCommandPacket` —— **这是客户端端执行任意指令的通道，如果权限漏判就是严重远程命令注入**。需逐个复核。

**统一修法模板**：

```java
public static void handle(Packet msg, Supplier<NetworkEvent.Context> ctx) {
    ctx.get().enqueueWork(() -> {
        ServerPlayer sender = ctx.get().getSender();
        if (sender == null) return;
        // 1. 权限门禁
        if (needsOp && !sender.hasPermissions(4)) return;
        // 2. 范围 clamp
        int lvl = Mth.clamp(msg.spellLevel, 1, Spell.MAX_LEVEL);
        // 3. 持有 / 冷却校验
        if (!MagicData.getPlayerMagicData(sender).hasSpell(spellId)) return;
        // 4. 距离校验（对需要 target 的包）
        if (sender.distanceToSqr(target) > MAX_DIST_SQ) return;
        // 5. 业务逻辑
        ...
    });
    ctx.get().setPacketHandled(true);
}
```

### 3.3 包 ID 自增分配容易破坏向后兼容

`ModNetwork.register()` 中 32 个包全靠 `packetId++`。`PROTOCOL_VERSION = "3"` 说明已经 bump 过两次。

**建议**：在 `register()` 顶部加注释：
```java
// ⚠️ 包 ID 按注册顺序分配。新增的包只能追加到末尾，绝对不要在中间插入。
// 破坏时要把 PROTOCOL_VERSION 一起 bump。
```
