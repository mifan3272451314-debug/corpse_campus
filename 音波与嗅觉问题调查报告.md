# 音波 / 嗅觉渲染问题调查报告

## 一、问题概述

当前 [`音波`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 与 [`嗅觉`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 两个客户端可视化能力仍未达到可用状态。

截至本报告编写时，玩家侧实际反馈如下：

- **音波**：
  - 已经可以在部分场景中做到“隔墙看到目标”；
  - 但标记位置仍然存在**明显错位**；
  - 并且当前表现疑似存在**前后方向判断异常**，即“生物在背后时反而更正常”。
- **嗅觉**：
  - 玩家反馈仍然是**完全看不到任何有效显示**；
  - 现有实现即便已经多次调整，仍未形成稳定可见的脚印 / 轨迹表现。

## 三、音波问题调查结论

### 1. 已确认现象

- 玩家反馈音波已经能在部分场景中“看到目标”，但标记仍有明显错位；
- 当前实现**不是 HUD 投影**，而是由 [`AbilityClientHandler.onRenderLevelStage()`](src/main/java/com/mifan/spell/AbilityClientHandler.java) 在 `AFTER_TRANSLUCENT_BLOCKS` 阶段调用 [`SonicSenseClientHandler.renderLevel()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 进行世界空间渲染；
- [`SonicSenseClientHandler.renderOverlay()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 只负责轻量暗化与氛围覆盖，并不负责目标定位；
- 实际定位锚点取自 `target.getBoundingBox().getCenter()` 再叠加 `target.getBbHeight() * 0.15D`，随后用 `camera.getLeftVector()` / `camera.getUpVector()` 生成朝向相机的 billboard；
- 当前代码中已经**不存在** `worldToScreen()`、`screenPos.z` 过滤或其他 HUD 屏幕空间投影判断分支。

### 2. 当前实现路径

- [`SonicSenseClientHandler.tick()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 只维护监听状态和动画帧计数；
- [`SonicSenseClientHandler.renderLevel()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 会扫描范围内活体目标，并按距离和脉冲动态调整贴图尺寸与透明度；
- 每个音波标记都通过 [`SonicSenseClientHandler.addBillboardQuad()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 直接提交到世界空间顶点缓冲，而不是先映射到屏幕坐标；
- 当前链路里没有额外的网络同步，也没有单独的“前方 / 后方 HUD 指示器”逻辑。

### 3. 当前判断

音波问题尚未真正解决，原因大概率还在以下几类之一：

1. 应当围绕**现行 billboard 渲染分支**排查，而不是再追已经删除的 `worldToScreen()` / HUD 投影路径；
2. 错位更可能来自锚点选取、billboard 朝向计算、渲染阶段、深度行为或实体移动插值，而不是旧版的屏幕空间前后过滤；
3. “背后反而更正常”这一现象虽然值得继续验证，但当前代码里没有显式的前后裁剪逻辑，因此问题更可能出在世界空间渲染本身，而不是遗留的 `z` 判断；
4. 如果后续仍要保留当前架构，排查重点应该放在 [`SonicSenseClientHandler.renderLevel()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java) 和 [`SonicSenseClientHandler.addBillboardQuad()`](src/main/java/com/mifan/spell/SonicSenseClientHandler.java)。

### 4. 当前结论

> **音波目前仍属于“能部分显示，但世界空间定位仍不稳定”的状态；后续分析必须基于当前 billboard 路线，而不是已删除的 HUD / `worldToScreen()` 路线。**

## 四、嗅觉问题调查结论

### 1. 已确认现象

- 玩家反馈为“仍然看不到任何东西”；
- 当前客户端实现**不是粒子方案**，而是在 [`OlfactionClientHandler.renderLevel()`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 中手工提交 `POSITION_COLOR` 四边形脚印；
- [`OlfactionClientHandler.tick()`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 会在客户端本地扫描范围内低于 75% 血量的实体，并基于目标移动情况往 `FOOTPRINTS` 列表里追加脚印；
- 与此同时，服务端 [`AbilityEventHandler.updateOlfactionTrail()`](src/main/java/com/mifan/spell/AbilityEventHandler.java) 还在实体 `PersistentData` 中维护另一套轨迹数据，用于 [`tickOlfaction()`](src/main/java/com/mifan/spell/AbilityEventHandler.java) 的速度奖励判定；
- 当前网络层并没有任何嗅觉轨迹同步包，因此客户端渲染和服务端玩法效果并不共用同一份轨迹数据。

### 2. 当前实现路径

- [`OlfactionClientHandler.tick()`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 每 3 tick 扫描一次附近低血量实体；
- [`OlfactionClientHandler.captureFootprint()`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 仅在目标发生足够位移时，按 `xo/zo -> x/z` 插值结果生成本地脚印；
- [`OlfactionClientHandler.renderLevel()`](src/main/java/com/mifan/spell/OlfactionClientHandler.java) 再把这些脚印绘制成两层带透明度变化的地面四边形；
- [`AbilityEventHandler.updateOlfactionTrail()`](src/main/java/com/mifan/spell/AbilityEventHandler.java) 在服务端每 6 tick 记录一份最多 24 个点、100 tick 过期的轨迹；
- [`ModNetwork.register()`](src/main/java/com/mifan/network/ModNetwork.java) 当前没有注册任何嗅觉相关的 clientbound / serverbound 包。

### 3. 当前判断

嗅觉当前最大问题不是“颜色不够亮”，而是**整条可视化链路没有被验证跑通**。潜在原因包括：

1. 当前最大的结构性问题是**客户端显示与服务端玩法各自维护一套轨迹数据**，两者已经脱节；
2. 客户端侧脚印是否出现，取决于目标是否被客户端看见、是否处于低于 75% 血量状态、以及最近是否真的发生了足够位移；
3. 即使服务端轨迹数据维护完全正确，客户端现在也**拿不到**那份轨迹，因为 [`ModNetwork.java`](src/main/java/com/mifan/network/ModNetwork.java) 里没有对应同步链路；
4. 因此，单独继续调整颜色、透明度或地面贴片形状，并不能验证“嗅觉轨迹功能”是否真的跑通；
5. 后续需要先在“补服务端到客户端同步”与“彻底改成纯客户端推导方案”之间选定一个统一的数据源。

### 4. 当前结论

> **嗅觉目前的核心问题不是单纯的渲染参数，而是数据链路分裂且缺少同步闭环；在统一数据源之前，‘看不见’的问题很难靠单纯调渲染解决。**

## 五、与外部参考实现的对比结论

已参考外部项目中“侦探听音定位”相关代码，核心结论如下：

- 参考实现的“隔墙显示”依赖的是 **HUD 投影**，而不是世界空间贴片；
- 参考实现显示的是**声音事件位置**，不是持续绑定在实体模型上的 3D 标记；
- 当前仓库中的**音波**实现走的是世界空间 billboard 路线，**嗅觉**实现走的是客户端本地脚印四边形路线；
- 因此外部参考更适合作为“是否要改架构”的参考，而不能直接拿来解释当前代码为什么出问题。

这意味着：

- **音波**如果继续沿用当前 billboard 路线，就应优先解决世界空间锚点、朝向和渲染阶段带来的错位问题；
- **嗅觉**如果继续走脚印路线，就应先把服务端轨迹与客户端渲染统一到同一条数据链路上，再讨论视觉样式细节；
- 如果后续决定转回 HUD / 事件点路线，应明确视为一次架构调整，而不是继续沿用当前文档中的旧假设。
