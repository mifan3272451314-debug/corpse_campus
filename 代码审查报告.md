## 0. 执行摘要（TL;DR）

工程目前完成了 5 个流派 + **33 个法术**（35 槽位缺 1 条注册行）的一/二/三级全面落地，`AbilityRuntime` 作为核心门面已缩减到 **732 行、73 个 `TAG_*`、50+ public static 方法**（比初版略降，部分运行时已拆出 `NecromancerRuntime` / `ElementalDomainRuntime` / `MimicRuntime` / `GrafterRuntime` / `DominanceRuntime` 等专类）。整体架构清晰，ISS API 使用姿势基本规范。**主要风险集中在四个面**：

1. **P0 Bug**：东岳 A 级 `impermanence_monk` 全链路就绪（类、Runtime、MobEffect、lang、图标），但 `ModSpells.java` 漏掉一行注册，导致该异能实际无法发放。
2. **安全面**：32 个数据包中有约 5 条 serverbound 包对客户端字段零校验（距离、数值范围、权限），可被伪造。`AdminExecuteCommandPacket` 需严格 OP 门禁。
3. **可维护面**：`AbilityRuntime` 仍是典型 god class，73 个 `TAG_*` 常量仍集中在一处；主类 `corpsecampus.java` 残留 `EXAMPLE_BLOCK` / `EXAMPLE_ITEM` 模板；`Config.java` 全模板字段。
4. **设计一致性偏离**：`DominanceSpell` 冷却 4s 与文档"30 分钟"不符；`InstinctSpell` 15% vs 文档 10%；`authority_grasp` 代码里又叫 `MysticAttendantRuntime` / `SpiritWormEntity`，**同一个法术三个名字**。

**本次增量新发现汇总到 [代码修复清单.md](./代码修复清单.md)**：1 条 P0 / 5 条 P1 / 9 条 P2/P3，共 15 项。

## 2. `AbilityRuntime` —— god class 专章

### 2.1 规模与职责

该类同时承担：
- NBT 键命名空间（100+ 个 `TAG_*`）
- 每个技能的运行时入口（`tickDominance`、`tickElementalDomain`、`tickRecorderOfficer`、`performDaiyueDash`、`performExecutionerWaveSlash` 等）
- 通用工具（`findTargetInSight`、`findLivingEntityById`、`clear`、UUID ListTag 读写）
- 内部状态机（`ElementalDomainState` 嵌套类 ~180 行）
- 全局静态集合（`ACTIVE_ELEMENTAL_DOMAINS`、`ELEMENTAL_DOMAIN_RESTORE_TICKS`、`DOMAIN_RANDOM`）

### 2.2 Critical / High 级问题

**[High] NBT UUID 序列化脆弱（line ~855-863, 972-995）**
`getDominatedMobs` / `containsUuid` 用 `ListTag<StringTag>` 存 UUID，每 tick `UUID.fromString` + 线性搜索，且第 990 行 `IllegalArgumentException` 被静默吞。建议改 `CompoundTag.putUUID` 原生 UUID 存储，并用缓存 `Set<UUID>` 代替线性扫描。

**[High] `tickDominance` 每 tick 写 NBT（line 972-995）**
每游戏刻：
1. 读 UUID 列表
2. 尝试 `level.getEntity()` 每个 UUID
3. 重新构建 cleaned ListTag
4. `data.put(TAG_DOMINANCE_MOBS, cleaned)` — **每 tick 改变 PersistentData** 触发持续脏标记

建议：只在"生物死亡事件"或"新增支配生物"时更新；tick 里仅读不写。

**[High] Daiyue 冲刺分步扫描（`performDaiyueDash` line ~1153-1161）**
`for i=1..steps`（可达 40+），每步独立 `getEntitiesOfClass`。应扫一次整段路径 AABB，再 `hitEntityIds` 去重。

**[Medium] 静态集合未同步（line ~162-163）**
tick 主线程访问安全；但若将来在 `SetMidasTouchTimer` 等 serverbound handle 里直接触碰这些集合且未 `context.enqueueWork`，会 CME。当前代码看上去都走 `consumerMainThread` 是安全的，但请在顶部加注释显式声明"tick-thread only"。

**[Medium] `tickRecorderOfficer` 维度解析（line ~410-417）**
`ResourceLocation.tryParse` / `ResourceKey.create` 失败时静默回退到当前维度，玩家不知为何未被传送。加一次 logger warn。

**[Medium] `Config`/方法缺 isClientSide 防御**
多个 public static 方法（`performDaiyueDash`、`triggerElementalistBurst`、`beginElementalDomain` 等）假设调用者已在服务端。上层（Spell 的 `onCast`）确实都有 `if (!level.isClientSide)` 包围，但工具方法内加一句 `level instanceof ServerLevel` 断言做防御性编程更稳。

**[Medium] `SpellRegistry.FIREBOLT_SPELL.get()` 等 ISS 常量硬编码（`castRegisteredElementalSpell`，line ~563-568）**
ISS 升级后若重命名会编译失败。影响可控（build 阶段就会暴露），Nit 级。

### 2.3 可维护性建议

把 `AbilityRuntime` 拆成（建议 P2 排期）：
- `runtime/DominanceRuntime.java`
- `runtime/ElementalDomainRuntime.java`（含 `ElementalDomainState`）
- `runtime/RecorderOfficerRuntime.java`
- `runtime/TelekinesisRuntime.java`
- `runtime/NecroticRuntime.java`
- `runtime/common/AbilityTags.java` —— 把 `TAG_*` 按技能分内部类
- `runtime/common/EntityQueries.java` —— `findTargetInSight` / `findLivingEntityById` / UUID helpers

拆分原则：NBT 键 + tick 方法 + 清理方法成组，一个技能一个文件。这样新加技能时只新增一个文件，不再改 god class。

## 3. 事件处理层

### 3.1 `AbilityEventHandler`

**事件总线确认**：`@Mod.EventBusSubscriber(modid = ...)` 默认 bus 为 `Bus.FORGE`，当前订阅的 `PlayerTickEvent` / `LivingHurtEvent` 等都在 FORGE 总线，**注解写法正确**（agent 报告中关于 "需要显式 bus=FORGE" 的结论为误报）。

**[High #8] `reviveNecroticCaster` 中 `player.removeAllEffects()`**
复活时把玩家身上的所有 `MobEffect` 全清，包括可能存在的增益。建议只移除 `NECROTIC_REBIRTH_ARMED` 和本次致死的 debuff（poison/wither 等），保留 buff。

**[Medium] 危机感知 tick AABB 扫描（`tickDangerSense`，line ~365-387）**
每 5 tick 跑两次 `getEntitiesOfClass`（一次 Mob、一次 Player）。可合并成一次 `getEntities` 按类型分流。半径 18+3L，L=5 时接近 30 格，在人多的服务器累积。

**[Medium] 元素领域 tick 内 AABB 后再手动 `distanceToSqr`（`tickElementalDomain`，line ~467-471）**
把距离检查塞到 predicate 里即可省掉一次迭代。

**[Low] `onLivingHurt` 中 INSTINCT 三段式逻辑混在一个方法里**
建议拆三个私有 `tryInstinctInvulnerable` / `tryInstinctDodge` / `tryInstinctLastStand`，顶层按顺序调用。

**[Low] `onLivingAttackDealt` 中 Mania 分支用 `setCanceled+手动 hurt`**
等效"忽略护甲"。需确认是否为设计意图，是则加注释；否则改为 `setAmount(amount*1.5F)` 走原流程。

### 3.2 `AbilityClientHandler`

**`@Dist.CLIENT` 注解正确**，客户端类隔离到位。

**[Medium] 数据包触发 `minecraft.setScreen` 未包 `minecraft.execute()`（`openDominanceTargetScreen` 等）**
`consumerMainThread` 已经保证在主客户端线程执行，所以当前代码不崩溃。**但 ISS/Forge 未来若改成 `workerPool` 需要重新审查**——现在不改也行，P2 级。

**[Medium] 客户端静态集合（`DANGER_ENTITY_MARKS`、`SONIC_SOUND_PINGS`）切换世界不自动清理**
`clearClientMarks` 存在但找不到调用点。建议在 `ClientTickEvent` 里检测 `minecraft.level == null` 或 `player == null` 时调用一次。

**[Low] `RenderLevelStageEvent` 内的 `RenderSystem.*` 调用**
agent 报告建议用 `RenderSystem.recordRenderCall`，但 `RenderLevelStageEvent` 本身就派发在主渲染线程，直接调 `RenderSystem.*` 是 Forge 文档允许的做法。**误报，保持现状即可**。

**[Low] `drawDangerOverlay` 访问 `level.getEntity(entityId)`（line ~588-591）**
ClientLevel 的实体视图可能已被回收返回 null；当前代码有 `instanceof LivingEntity livingEntity && livingEntity.isAlive()` 保护但不 remove 过期 entry，长时间会积累。建议 iterator.remove。

## 5. 网络层

### 5.1 `ModNetwork`

**[High #5] packetId 自增顺序敏感**——已在 P0 清单。可以维持自增但必须在 `register()` 开头注释"永远追加到末尾，不得中插"。

**[Medium] `PROTOCOL_VERSION="1"` 硬编码**——长期不变可接受；一旦要做向后兼容就需要协议协商机制。

### 5.2 Clientbound（服务器→客户端）

- `DangerSensePingPacket` / `InstinctProcPacket` / `OpenDominanceScreenPacket` / `OpenMidasTouchScreenPacket`：encode/decode 对称，基本类型无风险。
- `OpenRecorderOfficerScreenPacket`：**[Critical #4]** `readUtf(32767)`，且发送端未截断 `targetName`。改 `readUtf(64)` + encode 前 `substring(0,64)`。

### 5.3 Serverbound（客户端→服务器）

**此段是全报告最关键的安全改动**，详见 P0 #1-3。当前状态：

```java
// SetMidasTouchTimerPacket.handle 摘要
if (sender != null) {
    MagicData magicData = MagicData.getPlayerMagicData(sender);
    if (MidasTouchSpell.consumeMidasMana(sender, magicData, packet.powerLevel)) {
        MidasBombRuntime.armFromPlayerSelection(
            sender, Math.max(1, packet.spellLevel),
            packet.timerSeconds, packet.powerLevel);  // ← 完全信任客户端
    }
}
```

客户端可伪造 `powerLevel=99`、`timerSeconds=Integer.MAX_VALUE`、`spellLevel=999`。`armFromPlayerSelection` 内部的 clamp 只对 `timerSeconds` 生效，其它字段直接往下传。修复模板：

```java
if (packet.spellLevel < 1 || packet.spellLevel > MidasTouchSpell.MAX_LEVEL) return;
if (packet.powerLevel < 1 || packet.powerLevel > 5) return;
packet.timerSeconds = MidasBombRuntime.clampSeconds(packet.timerSeconds);
// 再考虑检查 sender 是否真的持有 midas_touch 法术、法力足够、冷却 OK
```

`SetDominanceTargetPacket` 额外需要：
- 距离检查（同维度，`distanceToSqr` < 配置上限）
- 施法者当前有效 dominance 标记检查
- 防重入 / 冷却
