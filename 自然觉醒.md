## 1. 背景与定位

### 1.1 为什么加这条通道
当前觉醒只有一条路径:拿到 B 级异常特性吞噬物 → 右键使用
([AnomalyTraitItem.java:44-68](corpse_campus/src/main/java/com/mifan/item/AnomalyTraitItem.java#L44-L68))。
这条路径过于依赖"掉落 / 捡物",玩家无法通过**玩法行为本身**进入序列。
自然觉醒是一条平行通道:玩家在世界里完成特定玩法动作(放告示牌、平砍怪、
长时间蹲伏等)达成阈值后,由系统**主动**把对应的 B 级法术直接装进他的
异常法术书并完成觉醒。

### 1.2 定位
- **只在 B 级层触发**。A / S 级晋升机制保持不变
- **和吞噬 B 是互斥的**:任一通道先完成 → 该玩家就锁了主序列,另一通道不再生效
- **不影响嫁接师**:圣祈 A 级「嫁接师」的跨序列能力按原逻辑运作
- **非目标**:不改 40 上限本身、不改死亡重置语义、不动 ISS 原生 `SpellBook`

## 4. 数据结构

### 4.1 玩家持久化 NBT(挂在 `player.getPersistentData()` → `AnomalyP0` 下)

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `NaturalAwakeningProgress` | `CompoundTag` | 每条原子条件对应一个 key → 累计值 |
| `NaturalAwakeningDone` | `boolean` | 该玩家是否**通过自然觉醒**进入序列 |

#### 4.1.1 `NaturalAwakeningProgress` 的键设计
每个 key 对应一条原子条件(不对应每个法术,避免重复累计):

- `sign_place_count` (int) — 印记
- `block_break_count` (int) — 耐力
- `farmland_till_count` (int) — 沃土
- `bonemeal_crop_count` (int) — 宁禾(催熟作物)
- `melee_attack_count` (int) — 躁狂
- `hit_by_monster_count` (int) — 本能(玩家被怪物打中次数)
- `monster_kill_count` (int) — 冥化
- `craft_count` (int) — 岱岳 / 亲和 / 支配 / 嗅觉 / 愈合 / 药师(暂不实现)
- `crouch_tick` (long) — 危机子条件 a(5 分钟)
- `crawl_tick` (long) — 危机子条件 b(120 秒)
- `wall_touch_tick` (long) — 磁吸
- `fall_burst_max` (int) — 万象。一次性坠落最大值(不是累加,是取历史最大单次)
- `dungeon_clear_count` (int) — 音波 / 日光 / 念力 / 回魂(暂不实现)

#### 4.1.2 `NaturalAwakeningDone` 的用途
与现有 `BOOK_AWAKENED` 相互补:
- 吞噬 B 通道觉醒:`AnomalyAwakened = true`,**不**写 `NaturalAwakeningDone`
- 自然觉醒通道:**同时**写 `AnomalyAwakened = true` 和 `NaturalAwakeningDone = true`
- 两个标志在死亡时一起清零

### 4.2 不新增 SavedData
- 40 上限仍由 [AnomalyLimitService](corpse_campus/src/main/java/com/mifan/anomaly/AnomalyLimitService.java) 管理
- 不引入"待觉醒队列"等中心化结构,避免两套状态漂移

## 5. 触发流程

### 5.1 核心伪代码

```
每类事件到达时:
  ServerPlayer player = 事件主角

  # 0) 前置:已觉醒玩家的进度完全冻结
  if getBook(player).isAwakened():
      return

  # 1) 更新原子条件计数
  NaturalAwakeningProgress[condKey] += increment

  # 2) 检查是否达到阈值
  if not reachedThreshold(condKey, NaturalAwakeningProgress[condKey]):
      return

  # 3) 取该条件对应的法术 ID 与流派 ID
  (schoolId, spellId) = resolveSpellByCondition(condKey, player)
  if spellId == null:
      return    # 同文字多法术的细化待后续

  # 4) 40 席位满员时不觉醒,进度保留,席位释放后下次触发再检查
  if AnomalyLimitService.isCapReached():
      log("自然觉醒被 40 上限拦截:" + player.name)
      return

  # 5) 执行觉醒(等价于 tryAbsorbBTrait 的后半段,但 spellId 定向)
  awakeNaturally(player, schoolId, spellId)

awakeNaturally(player, schoolId, spellId):
  book = ensureBookPresent(player)
  if book.isAwakened(): return   # 并发保护

  addSpell(player, book, spellId, level=1, spellLevel=1)
  writeSequenceBinding(book, schoolId, AnomalySpellRank.B)
  book.tag["NaturalAwakeningDone"] = true
  AnomalyLimitService.markAwakened(player.UUID)
  updateBookSnapshot(player, book)
  refreshCurioState(player)
  player.displayClientMessage(translatable("message.corpse_campus.awakened.natural", 
                                            schoolName, spellZhName))
```

### 5.2 `tryAbsorbBTrait` 的复用方式
实现层有两种等价做法,本文档**不强制**哪一种:

- 做法 A:给 `tryAbsorbBTrait` 加 `ResourceLocation forcedSpellId` 参数,非 null 时跳过 candidates 随机段
- 做法 B:从 `tryAbsorbBTrait` 抽出一个包级方法 `applyAwakening(player, schoolId, spellId)`,两条通道共享

做法 B 更干净,推荐,但这是实现细节,留给后续决定。

### 5.3 `resolveSpellByCondition` 规则
- 输入:原子条件 key + 当前玩家
- 输出:`(schoolId, spellId)` 或 null
- **1:1 精确映射**:绝大多数原子条件只对应一个法术,直接返回
- **多对一情况(同文字多法术)**:§10 待细化;在规则确定之前 resolve 返回 null,即该玩家虽然达成阈值但不自动觉醒,需要吞噬 B 或等细化后补发

## 7. 40 席位与上限行为(关键口径)

### 7.1 最终口径(用户 2026-04-23 确认)

| 行为 | 未达上限 | 达到上限(40/40) |
| --- | --- | --- |
| 发放异常法术书 | ✓ 正常发 | **✓ 正常发**(不受上限影响) |
| B 级吞噬物掉落 | ✓ 正常掉 | **✓ 正常掉**(不阻止) |
| 吞噬 B 通道(右键吞噬物) | ✓ 正常触发 | ✗ 返回 `anomaly_cap_reached` 失败 |
| 自然觉醒通道 | ✓ 正常触发 | ✗ 不执行觉醒,进度保留 |
| 自然觉醒进度累加 | ✓ 正常累加 | **✓ 仍然累加**(不因满员而停) |
| 已觉醒玩家的进度累加 | — | ✗ 冻结(玩家维度) |

### 7.2 进度冻结范围
- 只冻结**觉醒者本人**的 `NaturalAwakeningProgress`(所有子计数器)
- 其他玩家不受影响,世界维度照常运行

### 7.3 满员时的"阻塞等待"语义
- 玩家 A 达成条件但正好卡在满员 → 不觉醒,`NaturalAwakeningProgress[condKey]` 保留已达到的值
- 任意玩家死亡释放一个席位后,玩家 A 的**下一次**相关事件(哪怕只增加 1)会重新触发 §5.1 流程的 check,此时 `isCapReached() == false`,觉醒通过
- 不引入中心化"待觉醒队列" / 不在 tick 里主动扫描

## 8. 死亡重置

### 8.1 现有行为(已实装)
[onPlayerDeath](corpse_campus/src/main/java/com/mifan/anomaly/AnomalyEventHandler.java#L72-L99)
已经在玩家死亡时执行:
- `clearBookAndCollectTraitDrops`:清空书内法术、清 `AnomalyAwakened / MainSequence / HighestRank`
- `AnomalyLimitService.clearAwakened`:释放 40 席位
- 生成历史阶段物品并放进 `PendingTraitDrops` 等待 `onPlayerDrops` 落地

### 8.2 自然觉醒需要追加的清理
在同一个 `onPlayerDeath` 钩子里**追加**:
- 清 `NaturalAwakeningDone`
- 清 `NaturalAwakeningProgress`(**全量清零**,不保留任何条件的历史累计)

### 8.3 为什么死后进度全清
- 和吞噬 B 通道对称:死亡本来就会清掉所有序列状态,自然觉醒的"进度条"也是序列前置状态,一并清掉
- 防止"死亡前接近阈值 → 重生后 1 个动作就觉醒"的刷阈值漏洞
- 用户 2026-04-22 口径:"玩家死亡后会洗掉觉醒效果以及相关序列和位阶",语义自然延伸到进度

## 11. 代码对齐口径的一次性清理(2026-04-23 已完成)

本文档 §7 明确:**40 上限只拦觉醒通道,不拦发书、不拦掉落**。为对齐这一口径,
随本文档一次性完成了以下代码清理(不影响 §1-§10 的设计与后续实现):

### 11.1 去掉满额时的 B 级吞噬物掉落拦截

- **原行为**:`AnomalyEventHandler.onPlayerDrops` 里的 `blockBDrops` 分支会在
  `globalCapEnabled && disableBDropWhenFull && isCapReached()` 三条都满足时跳过 B 级掉落
- **新行为**:B 级吞噬物在任何情况下(含满 40)都照常掉落;玩家捡到后是否能
  吞噬 / 觉醒仍由 `tryAbsorbBTrait` 的 `isCapReached()` 检查决定(§7.1)
- **改动面**:
  - [AnomalyEventHandler.java](corpse_campus/src/main/java/com/mifan/anomaly/AnomalyEventHandler.java#L101-L126):
    `onPlayerDrops` 移除 `blockBDrops` 变量与 if 分支,附带删掉未用的 `ModItems` import
  - [AnomalyConfig.java](corpse_campus/src/main/java/com/mifan/anomaly/AnomalyConfig.java):
    `DISABLE_B_DROP_WHEN_FULL` spec、`disableBDropWhenFull` 字段与 `onLoad` 加载行全部删除
  - [MagicCommand.java:603](corpse_campus/src/main/java/com/mifan/command/MagicCommand.java#L603):
    `/magic config show` 中对应的展示行删除
  - [法术书要求.md](./法术书要求.md) §14 配置总表与示例 toml 中的 `disableBDropWhenFull` 行删除

### 11.2 对历史 config 文件的影响

旧服务端的 `config/corpse_campus-common.toml` 里若残留 `limit.disableBDropWhenFull = true`,
Forge 会在加载时直接忽略未知键,不报错、不需要迁移。
